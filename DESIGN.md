## Problem Statement

In Toronto/GTA, Google Maps "currency exchange" results are polluted in three specific zones: generic city-wide searches (6/10 first-page results are real walk-in shops), Markham (5/10), and downtown office towers (7/10). 12.8% of all listings (35 of 274 measured) are online-only businesses with no walk-in counter — a real user travelled to one and walked out with cash unexchanged. No site on the SERP answers the one question that matters: **does this place physically take your cash at a counter?**

## Demand Evidence

- One observed failing user (Kim's friend, HK newcomer — travelled to an online-only "currency exchange," real consequence)
- 35 verified trap listings in a 274-place scrape of real Maps results (2026-06-03, $0, Playwright)
- 5+ RedFlagDeals "best rate Toronto" threads spanning 2017-2026 — proven recurring demand for the *generic* query
- HONEST TIER (D2): demand for cash-verification *specifically* is a hypothesis. Nobody types "currency exchange that takes cash." The site must win the generic neighbourhood query and deliver verification as the differentiator on-page.

## Status Quo

Suburban Maps search already works (North York 10/10, Scarborough 9/10, Mississauga 9/10 first-page accuracy). SERP for "best currency exchange toronto": single-shop marketing pages + Wise/Exiap affiliate listicles covering 3-10 shops. No full directory. Two predecessor apps dead at zero ratings (2016, 2020). 51.ca = converter + paid ads, not a directory.

## Target User & Narrowest Wedge

**The newcomer with cash errands** (D3) — the only observed failing persona. Corridors: Markham, Scarborough, North York, plus the downtown trap zone. Wedge: per-neighbourhood pages ("currency exchange in X") whose data carries an honest tri-state status no competitor has:
- `walk-in confirmed` (with evidence + date)
- `online-only / no cash` (excluded from listings, exposed on a dedicated trap page)
- `unverified — call ahead` (honest default)

## Constraints

- Solo founder, $0 budget, no network. WSL box available for cron. GitHub (kimwwk) authenticated; Cloudflare needs one-time `wrangler login`.
- Backend must be **automated and repeatable from this machine** (founder directive): one command refreshes the data; the site only shows what the pipeline can reach.
- Rates are NOT v0 (Toronto shops are call-only for rates).
- Apify: not used in v0 (no token; local Playwright scraper already proven on 392 rows). Documented as fallback if Google blocks local scraping.

## Premises (agreed D4, premise 2 revised after founder directive)

1. The fight is the generic query "currency exchange [neighbourhood]"; the win is the data.
2. **REVISED:** Automation is the refresh/monitoring layer; verification is a layered claim. The pipeline auto-classifies with evidence; the phone-verified tier exists in the schema from day 1 and is filled by a one-afternoon call pass over the ambiguous tail (~30-50 shops), not recurring labor. The site never claims more than its evidence tier.
3. Built for the newcomer with cash errands; corridor neighbourhoods first; hours/phone/walk-in status above rates.
4. Scoreboard: Search Console organic clicks + repeat visits. No community metric.
5. The problem is small, real, boring — built because it is cheap to win and the data asset compounds.

## Cross-Model Perspective

Independent cold read (Claude subagent, fresh context): sided with original premise 2 — "the pipeline that produced the bad data (Maps) cannot be the pipeline that certifies the good data"; predicted automated 'takes cash=yes' precision of 60-75% on the ambiguous tail, with misses in the dangerous direction (false yes). Proposed test: auto-classify the tail, then phone-verify all of it; if precision <90%, unverified badges are a liability. Reconciliation adopted: automation refreshes, evidence tiers gate claims, calls upgrade the tail. ## Approaches Considered

### Approach A: Static site + local pipeline (CHOSEN — minimal viable)
Reuse the proven Playwright scraper → dedupe → rule-based classifier with evidence strings → `data/shops.json` → custom Node static-site generator → Cloudflare Pages. No framework, no server, no DB. Effort: S. Risk: Low. Reuses: scraper from Test 0, classification methodology from maps-gap-analysis.md, FINTRAC CSV join.

### Approach B: Ideal architecture
Same pipeline + SQLite + admin review queue + verification-ledger UI + per-shop change history. Effort: M. Risk: Med. Deferred: build A first; B's schema fields (status, evidence, verified_date, source) are baked into A's JSON so migration is additive.

### Approach C: Data-as-content (lateral)
Pipeline auto-publishes a monthly "State of GTA currency exchange" report + the trap exposé as the SEO/content engine; directory generated from the same data. Partially adopted: the trap page ("listings that won't take your cash") ships in v0 as a content asset.

## Recommended Approach

A, with B's data schema and C's trap page. Rationale: fastest to deployed truth, $0, every component already proven in this console, and the founder's directive ("show data we can reach in this console") is satisfied honestly via evidence-tiered statuses.
